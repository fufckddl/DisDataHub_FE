# South Korea Population and Regional OpenAPI Sources

_Last reviewed: 2026-05-28. Scope: South Korea / Republic of Korea (남한) sources suitable for a dashboard. “OpenAPI” below means Korean public-sector open API/data-service listings, not necessarily an OpenAPI Specification YAML/JSON contract._

## Recommended source stack

1. **Primary population counts:** Ministry of the Interior and Safety (MOIS, 행정안전부) resident-registration population datasets on data.go.kr.
2. **National statistical context and historical series:** KOSIS shared OpenAPI.
3. **Map-ready spatial joins:** SGIS OpenAPI for census/geostatistical data and VWorld/NSDI administrative-boundary WMS/WFS.
4. **Local/high-frequency context:** Seoul Open Data Plaza living-population feeds where Seoul is in scope; KMA weather and MOIS safety/disaster APIs for contextual layers.

## Integration notes

- **Code strategy:** Use official administrative/legal codes as the join key wherever possible. Korean local-government names change and can collide; names should be display labels, not primary keys.
- **Caching:** Cache raw API responses by source, endpoint, parameter hash, and reference period. Population snapshots should be immutable once ingested.
- **Auth:** Most data.go.kr, VWorld, SGIS, KOSIS, and Seoul Open Data endpoints require a service key. Keep keys server-side or in environment-managed proxy endpoints; do not commit keys to the frontend.
- **Cadence labels:** Display the source reference period separately from ingestion time. “Real-time” portal metadata can describe API availability while the underlying resident-register statistic is monthly or annual.
- **Privacy:** Avoid rendering extremely fine tong/ban or census-area statistics without aggregation, suppression, or clear public-data provenance. Prefer sigungu/eupmyeondong views for public dashboard defaults.
- **Licensing:** Check each dataset’s current license before release. Many data.go.kr datasets are free/public-use, but some require source attribution or include third-party rights notices.

## Suggested implementation order for this dashboard

1. Build a source registry for MOIS monthly resident-registration population and KOSIS statistical tables.
2. Add a shared administrative-code dimension table and boundary loader from VWorld/SGIS.
3. Add optional contextual layers: KMA weather, MOIS safety information, and Seoul living population for Seoul-only views.
4. Add source badges in the UI: provider, reference month/year, update cadence, and license/attribution note.
